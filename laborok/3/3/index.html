    3. feladat – Medallion Lakehouse Delta Lake-kel¶

    A modern Lakehouse architektúra kombinálja az adattó (Data Lake) rugalmasságát és az adattárház (DWH) megbízhatóságát. Az adatokat nyílt formátumban (Parquet fájlokban) tároljuk, amelyek közvetlenül lekérdezhetők – nincs szükség költséges DWH szerverre. A Delta Lake egy nyílt forráskódú tárolási formátum, amely a sima Parquet fájlok fölé épít ACID tranzakciós réteget, sémaellenőrzést és time travel képességet.

    Delta Lake vs sima Parquet

    Képesség Sima Parquet Delta Lake
    ACID tranzakciók Nincs Igen – _delta_log/ JSON naplóval
    Sémaellenőrzés Nincs schema_mode='enforce'
    Time travel Nincs version_as_of=N vagy timestamp_as_of
    Upsert (MERGE) Manuálisan Beépített .merge().execute()
    Kis fájlok összevonása Manuálisan OPTIMIZE paranccsal

    A _delta_log/ mappa JSON fájlokat tartalmaz: minden írási művelet egy új 0000N.json naplóbejegyzést hoz létre, amely pontosan leírja, mi változott – pontosan mint egy SQL tranzakciós napló.

    A három réteg:
    * Bronze – nyers, változatlan forrásadat. Soha nem módosítjuk, csak hozzáírunk. Auditálási és visszakeresési alap.
    * Silver – megtisztított, típusos, sémaellenőrzött adat. Idempotensen újragenerálható a Bronze-ból.
    * Gold – aggregált, üzleti logikával feldolgozott adat. BI eszközök, ML modellek alapja.

    Bronze réteg – nyers adatok Delta táblába¶

    1. A második cellában hozzuk létre a könyvtárstruktúrát és töltsük fel a Bronze réteget:

      import pandas as pd
import duckdb
import pyarrow as pa
from deltalake import DeltaTable, write_deltalake   # Delta Lake olvasás és írás
from sqlalchemy import create_engine
import os, json

LAKEHOUSE = "/home/jovyan/work/lakehouse"   # Lakehouse gyökérmappa

# A három réteg könyvtárainak létrehozása
for path in ['bronze/orders', 'bronze/customers', 'bronze/products',
             'silver/orders', 'gold/revenue']:
    os.makedirs(f"{LAKEHOUSE}/{path}", exist_ok=True)

engine = create_engine("postgresql://labor:labor@postgres:5432/oltp_source")

# Bronze: NYERS adatok – változtatás nélkül, ahogy a forrásból jön
# Szándékosan nem szűrünk és nem tisztítunk – a Bronze az "igazság egyetlen forrása"
df_orders_raw = pd.read_sql("""
    SELECT
        oi.id       AS item_id,
        o.id        AS order_id,
        o.customer_id,
        oi.product_id,
        o.order_date::TEXT AS order_date,    -- DATE → string, hogy a schema rugalmas legyen
        oi.quantity,
        oi.unit_price::FLOAT AS unit_price,
        o.status,
        NOW()::TEXT AS _ingested_at           -- Technikai mező: mikor töltöttük be
    FROM order_items oi
    JOIN orders o ON oi.order_id = o.id
""", engine)

df_customers_raw = pd.read_sql(
    "SELECT id, name, email, city, country, created_at::TEXT FROM customers", engine
)
df_products_raw = pd.read_sql(
    "SELECT id, name, category, unit_price::FLOAT AS unit_price FROM products", engine
)

print(f"Bronze betöltendő sorok:")
print(f"  orders+items: {len(df_orders_raw):,}")
print(f"  customers:    {len(df_customers_raw):,}")
print(f"  products:     {len(df_products_raw):,}")

# Bronze írás Delta táblába – mode='overwrite': újrafuttatáskor felülírja a korábbi verziót
write_deltalake(f"{LAKEHOUSE}/bronze/orders",    df_orders_raw,    mode="overwrite")
write_deltalake(f"{LAKEHOUSE}/bronze/customers", df_customers_raw, mode="overwrite")
write_deltalake(f"{LAKEHOUSE}/bronze/products",  df_products_raw,  mode="overwrite")

# Delta napló megtekintése – az ACID tranzakciók naplója
history = DeltaTable(f"{LAKEHOUSE}/bronze/orders").history()
print("\nBronze/orders Delta tranzakciónapló:")
for entry in history:
    # Minden write_deltalake() hívás egy új bejegyzést hoz létre
    print(f"  verzió {entry['version']}: {entry['operation']} – {entry['timestamp']}")

# _delta_log tartalmának bemutatása
log_dir = f"{LAKEHOUSE}/bronze/orders/_delta_log"
log_files = sorted(os.listdir(log_dir))
print(f"\n_delta_log fájlok: {log_files}")
# A Delta log fájlok NDJSON formátumúak: soronként egy JSON akció (metaData, add, stb.)
# json.load() az egész fájlt egy objektumnak várja – json.loads(sor) kell soronként
with open(f"{log_dir}/{log_files[0]}") as f:
    actions = [json.loads(line) for line in f if line.strip()]
print(f"Első napló bejegyzés akciói: {[list(a.keys())[0] for a in actions]}")

      Miért strings a order_date a Bronze-ban?

      A Bronze rétegben szándékosan string típusban tároljuk a dátumot. Ha a forrásrendszer egyszer '2024-01-15', máskor '15/01/2024' formátumban küld dátumot, a Bronze mindkettőt képes fogadni. A típuskonverzió a Silver réteg feladata – ez az elválasztás teszi lehetővé a teljes auditálhatóságot.

    Silver réteg – adattisztítás és sémaellenőrzés¶

    1. A harmadik cellában valósítsuk meg a Silver réteget explicit sémával:

      # Silver séma definitíció PyArrow-val – minden mezőnek pontos típusa van
# Ha a Bronze adat nem felel meg ennek, az írás hibával leáll
silver_schema = pa.schema([
    pa.field("item_id",      pa.int64()),
    pa.field("order_id",     pa.int64()),
    pa.field("customer_id",  pa.int64()),
    pa.field("product_id",   pa.int64()),
    pa.field("order_date",   pa.date32()),      # string → dátum típus (Silver-ben már DATE)
    pa.field("quantity",     pa.int32()),
    pa.field("unit_price",   pa.float64()),
    pa.field("total_amount", pa.float64()),     # Levezetett mező: quantity × unit_price
    pa.field("status",       pa.string()),
])

# Bronze adatok beolvasása
df_bronze = DeltaTable(f"{LAKEHOUSE}/bronze/orders").to_pandas()
raw_count  = len(df_bronze)

# TODO: Valósítsuk meg a Silver tisztítást!
#
# Lépések:
#   1. NULL szűrés: dropna(subset=['customer_id', 'product_id', 'unit_price'])
#
#   2. Negatív értékek kiszűrése: query("quantity > 0 and unit_price > 0")
#
#   3. Típuskonverzió:
#      .assign(
#          order_date   = lambda x: pd.to_datetime(x['order_date']).dt.date,
#          total_amount = lambda x: (x['quantity'] * x['unit_price']).round(2),
#      )
#      (A .dt.date a datetime64 típusból Python date objektumot csinál – PyArrow date32-nek megfelelő)
#
#   4. Csak a silver_schema-ban definiált oszlopok maradjanak:
#      [['item_id', 'order_id', 'customer_id', 'product_id',
#        'order_date', 'quantity', 'unit_price', 'total_amount', 'status']]
#
# Utána:
#   - Írjuk Silver Delta táblába: write_deltalake(..., pa.Table.from_pandas(df_silver, schema=silver_schema), mode="overwrite", schema_mode="overwrite")
#   - Írjuk ki: Kiszűrt sorok száma és Silver sorok száma
df_silver = None   # TODO: cseréld le a tényleges implementációra
pass
      Megoldási útmutató – kattints a tippekért 
      df_silver = (df_bronze
    .dropna(subset=['customer_id', 'product_id', 'unit_price'])  # NULL kulcsok kiszűrése
    .query("quantity > 0 and unit_price > 0")                    # Érvénytelen értékek
    .assign(
        order_date   = lambda x: pd.to_datetime(x['order_date']).dt.date,
        total_amount = lambda x: (x['quantity'] * x['unit_price']).round(2),
    )
    [['item_id', 'order_id', 'customer_id', 'product_id',
      'order_date', 'quantity', 'unit_price', 'total_amount', 'status']]
)

write_deltalake(
    f"{LAKEHOUSE}/silver/orders",
    pa.Table.from_pandas(df_silver, schema=silver_schema),
    mode="overwrite",
    schema_mode="overwrite"   # Felülírja a séma definícióját is
)

print(f"Bronze → Silver tisztítás:")
print(f"  Nyers sorok:   {raw_count:,}")
print(f"  Silver sorok:  {len(df_silver):,}")
print(f"  Kiszűrt sorok: {raw_count - len(df_silver):,}")

      Miért schema_mode="overwrite"? Az első Silver íráskor még nem létezik a séma a Delta logban. Az "overwrite" mode lehetővé teszi, hogy felülírjuk – ha már egyszer le van rögzítve, a következő futásoknál "enforce"-ra érdemes váltani, hogy véletenül sem tudjuk megsérteni.

    2. A negyedik cellában mutassuk meg a sémaellenőrzés erejét egy szándékos hibával:

      # Sémavioláció demonstráció – szándékosan rossz típusú oszlop
df_wrong = df_silver.copy()
df_wrong['quantity'] = df_wrong['quantity'].astype(str)   # int32 helyett string!

try:
    write_deltalake(
        f"{LAKEHOUSE}/silver/orders",
        df_wrong,
        mode="append",
        schema_mode="enforce"     # Szigorú ellenőrzés: eltérés esetén kivétel
    )
    print("HIBA: nem kellett volna sikerülnie!")
except Exception as e:
    # A Delta Lake sémaellenőrzés megakadályozta a hibás adat betöltését
    print(f"Sémavioláció elkapva (várt hiba):")
    print(f"  {type(e).__name__}: {str(e)[:150]}")

      Miért fontos a sémaellenőrzés?

      Sémaellenőrzés nélkül egy hibás upstream adat (pl. string mennyiség egy CSV-ben) csendben bekerülhet az adatbázisba, és csak sokkal később, egy lekérdezésnél derül ki a hiba. A Delta Lake schema_mode="enforce" opciója azonnal hibát dob, és a teljes tranzakciót visszagörgeti – az adatbázis konzisztens állapotban marad.

    Gold réteg – aggregáció és upsert¶

    1. Az ötödik cellában hozzuk létre a Gold réteget DuckDB aggregációval:

      dcon = duckdb.connect()   # In-memory DuckDB – csak aggregációhoz, nem perzisztens

# Silver Delta tábla beolvasása pandas DataFrame-be, majd regisztrálás DuckDB-ben
# A DuckDB közvetlenül is tudna Delta-t olvasni, de a pandas út átláthatóbb
silver_df = DeltaTable(f"{LAKEHOUSE}/silver/orders").to_pandas()
dcon.register("silver_orders", silver_df)

# Gold: napi bevétel aggregáció
df_gold = dcon.execute("""
    SELECT
        order_date                          AS full_date,
        COUNT(DISTINCT order_id)             AS order_count,     -- Rendelések (nem tételek)
        COUNT(*)                             AS item_count,      -- Tételek száma
        ROUND(SUM(total_amount), 2)          AS total_revenue,
        ROUND(AVG(total_amount), 2)          AS avg_item_value
    FROM silver_orders
    WHERE status = 'completed'              -- Csak teljesített rendelések kerülnek a Gold-ba
    GROUP BY order_date
    ORDER BY order_date
""").df()

print(f"Gold aggregáció: {len(df_gold)} nap adata")
print(df_gold.head(5).to_string(index=False))

# Első Gold írás (version 0)
write_deltalake(f"{LAKEHOUSE}/gold/revenue", df_gold, mode="overwrite")
print("\nGold/revenue v0 megírva.")

#
# 4. Delta verziónapló kiírása:
#    for entry in DeltaTable(f"{LAKEHOUSE}/gold/revenue").history():

      Megoldási útmutató – kattints a tippekért 
      # 1. Frissítendő sorok
sample_dates = df_gold['full_date'].sample(3, random_state=1).tolist()
df_update = df_gold[df_gold['full_date'].isin(sample_dates)].copy()
df_update['total_revenue'] = (df_update['total_revenue'] * 1.05).round(2)
df_update['order_count']   = df_update['order_count'] + 1

# 2. MERGE: a target ('t') és source ('s') alias-okat a predicate-ben és az update/insert-ben is használjuk
gold_table = DeltaTable(f"{LAKEHOUSE}/gold/revenue")
(gold_table
    .merge(
        source=df_update,
        predicate="t.full_date = s.full_date",
        source_alias="s",
        target_alias="t"
    )
    .when_matched_update_all()
    .when_not_matched_insert_all()
    .execute()
)
print(f"Upsert kész – {len(df_update)} nap adatai frissítve")

# 3. Time travel összehasonlítás
df_v0 = DeltaTable(f"{LAKEHOUSE}/gold/revenue", version=0).to_pandas()
df_v1 = DeltaTable(f"{LAKEHOUSE}/gold/revenue").to_pandas()

test_date = sample_dates[0]
rev_v0 = df_v0[df_v0['full_date'] == test_date]['total_revenue'].values[0]
rev_v1 = df_v1[df_v1['full_date'] == test_date]['total_revenue'].values[0]
print(f"\nTime travel ({test_date}):")
print(f"  v0 (eredeti):    {rev_v0:,.2f}")
print(f"  v1 (frissített): {rev_v1:,.2f}  (+{rev_v1 - rev_v0:,.2f})")

# 4. Verziónapló
print("\nGold/revenue Delta verziónapló:")
for entry in DeltaTable(f"{LAKEHOUSE}/gold/revenue").history():
    print(f"  v{entry['version']}: {entry['operation']} – {entry['timestamp']}")

      Miért MERGE az egyszerű overwrite helyett? Az overwrite a teljes táblát újraírja – nagy tábláknál ez lassú és drága. A MERGE csak az érintett sorokat módosítja, a többi változatlan marad. Ugyanakkor ACID garantiát nyújt: ha a MERGE közben hiba lép fel, a tábla az eredeti állapotban marad (a _delta_log naplóba nem kerül bejegyzés).
