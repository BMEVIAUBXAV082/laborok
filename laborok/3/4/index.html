    4. feladat – dbt ELT pipeline¶

    A dbt (data build tool) az ELT (Extract → Load → Transform) paradigma transzformációs motorja. Szemben az ETL-lel, ahol a transzformáció még betöltés előtt történik, az ELT esetén az adatokat először nyers formában töltjük be, majd az adatbázison belül transzformáljuk SQL segítségével. A dbt SQL modelleket Git-verziókövethető fájlokként kezeli, a ref() macro automatikusan feloldja a függőségi sorrendet, a schema.yml pedig beépített adatminőség-tesztelést biztosít.

    dbt alapfogalmak

    • ref('model_name') – modell hivatkozás és függőség deklaráció egyben. A dbt ebből építi fel a végrehajtási DAG-ot: automatikusan tudja, melyik modellt kell először lefuttatni.
    • Materializáció – hogyan materializálódjon a modell az adatbázisban:
    • view: egy nézet, amely mindig friss adatot ad, de nem tárol adatot
    • table: előre kiszámolt tábla – gyorsabb lekérdezés, de tárhelyet igényel
    • incremental: csak az új/megváltozott sorokat dolgozza fel – nagy tábláknál hatékony
    • Seed – statikus CSV fájl, amelyet a dbt seed tölt be az adatbázisba (pl. referencia listák).
    • Generic test – YAML-ban deklarált adatminőség-ellenőrzés: not_null, unique, accepted_values, relationships.

    Projekt inicializálás és konfiguráció¶

    1. A második cellában inicializáljuk a dbt projektet és hozzuk létre a konfigurációs fájlokat:

        ["dbt", "init", "labor04_dbt", "--skip-profile-setup"],

        capture_output=True, text=True

    print(result.stdout or result.stderr)

    shutil.rmtree(example_dir)
    print("models/example/ törölve.")

# dbt_project.yml – a projekt fő konfigurációs fájlja
# Ez határozza meg a projekt nevét, a modell elérési útvonalait és a materializációs alapértékeket
# textwrap.dedent: a behúzás eltávolításával pontos YAML-t kapunk
dbt_project_yml = textwrap.dedent("""\
    name: 'labor04_dbt'
    version: '1.0.0'

    seed-paths:  ["seeds"]

# Ez külön fájlban van, hogy ne kerüljön be a Git repositoryba (jelszavak, elérési utak)

      target: dev

    Staging modellek¶

    1. A negyedik cellában hozzuk létre a staging modell fájlokat:

      staging_dir = f"{PROJECT_DIR}/models/staging"
os.makedirs(staging_dir, exist_ok=True)

# stg_customers.sql – dim_customer forrásból (1. feladatból)
# A staging modell feladata: normalizált névhasználat, egyszerű tisztítás
stg_customers_sql = textwrap.dedent("""\
    -- stg_customers: az ügyfél dimenzió staging rétege
    -- Forrás: dim_customer tábla (1. feladatban töltöttük fel)
    SELECT
        customer_id,                  -- Natural key megőrzése
        name    AS customer_name,     -- Átnevezés a DWH névkonvenció szerint
        city,
        country
    FROM dim_customer
    """)
with open(f"{staging_dir}/stg_customers.sql", "w") as f:
    f.write(stg_customers_sql)

# stg_products.sql – dim_product + product_categories seed JOIN-nal
# A ref() macro automatikusan feloldja, hogy a product_categories seed előbb fut
stg_products_sql = textwrap.dedent("""\
    -- stg_products: termék dimenzió, kategória referenciával gazdagítva
    SELECT
        p.product_id,
        p.product_name,
        p.category,
        p.unit_price,
        pc.category_group,                    -- Seed-ből jön: Tech, Otthon, Iroda, FMCG, Szabadidő
        CAST(pc.is_seasonal AS BOOLEAN) AS is_seasonal  -- CSV-ben string volt, itt konvertálunk
    FROM dim_product p
    LEFT JOIN {{ ref('product_categories') }} pc ON p.category = pc.category
    -- LEFT JOIN: ha egy kategória nincs a seed-ben, a termék akkor is megjelenik (NULL category_group)
    """)
with open(f"{staging_dir}/stg_products.sql", "w") as f:
    f.write(stg_products_sql)

# stg_orders.sql – a ténytáblából staging, csak completed státuszú tételek
stg_orders_sql = textwrap.dedent("""\
    -- stg_orders: rendelési tételek staging – csak teljesített rendelések
    -- Forrás: fact_order_item (1. feladatban töltöttük fel)
    SELECT
        order_item_key,
        customer_key,
        product_key,
        order_date,
        quantity,
        unit_price,
        total_amount
    FROM fact_order_item
    WHERE total_amount > 0    -- Biztonsági szűrő: negatív összeg nem kerülhet a modellbe
    """)
with open(f"{staging_dir}/stg_orders.sql", "w") as f:
    f.write(stg_orders_sql)

# schema.yml – staging modellek dokumentációja és tesztjei
staging_schema = textwrap.dedent("""\
    version: 2

    models:
      - name: stg_customers
        description: "Ügyfél dimenzió staging réteg – normalizált névhasználattal"
        columns:
          - name: customer_id
            description: "Forrásrendszer ügyfél azonosítója (natural key)"
            tests: [not_null, unique]
          - name: customer_name
            tests: [not_null]

      - name: stg_products
        description: "Termék dimenzió staging réteg – kategória referenciával gazdagítva"
        columns:
          - name: product_id
            tests: [not_null, unique]
          - name: unit_price
            tests: [not_null]

      - name: stg_orders
        description: "Rendelési tételek staging – csak pozitív összegű sorok"
        columns:
          - name: order_item_key
            tests: [not_null, unique]
          - name: total_amount
            tests: [not_null]

with open(f"{staging_dir}/schema.yml", "w") as f:
    f.write(staging_schema)

print("Staging modellek és schema.yml létrehozva.")

    Mart modellek¶

    1. Az ötödik cellában hozzuk létre a mart_daily_revenue.sql modellt:

      marts_dir = f"{PROJECT_DIR}/models/marts"
os.makedirs(marts_dir, exist_ok=True)

# mart_daily_revenue.sql – napi bevétel összesítés
# Ez a modell aggregál: nem tárol sorokat, hanem napszintű összesítéseket
mart_daily_sql = textwrap.dedent("""\
    -- mart_daily_revenue: napszintű bevétel összesítés
    -- Grain: 1 sor = 1 nap összes teljesített rendelésének összesítése
    SELECT
        o.order_date                           AS full_date,
        COUNT(o.order_item_key)                AS item_count,         -- Tételek száma
        COUNT(DISTINCT o.customer_key)         AS unique_customers,   -- Hány különböző ügyfél rendelt
        ROUND(SUM(o.total_amount), 2)          AS total_revenue,
        ROUND(AVG(o.total_amount), 2)          AS avg_item_value      -- Átlagos tételérték
    FROM {{ ref('stg_orders') }} o
    GROUP BY o.order_date
    ORDER BY o.order_date
    """)
with open(f"{marts_dir}/mart_daily_revenue.sql", "w") as f:
    f.write(mart_daily_sql)

print("mart_daily_revenue.sql létrehozva.")

    2. A hatodik cellában valósítsuk meg a mart_customer_ltv.sql modellt – ez a hallgató önálló feladata:

      # TODO: Hozzuk létre a mart_customer_ltv.sql fájlt!
#
# Ez a modell az ügyfelek „lifetime value" (teljes bevétel) összesítését számítja.
#
# A modell felépítése:
#   - Forrás táblák: stg_orders (o) JOIN stg_customers (c) ON o.customer_key = c.customer_id
#   - GROUP BY: c.customer_id, c.customer_name, c.city, c.country
#   - Számított mezők:
#       COUNT(o.order_item_key)        AS total_orders
#       ROUND(SUM(o.total_amount), 2)  AS lifetime_value
#       ROUND(AVG(o.total_amount), 2)  AS avg_order_value
#       MIN(o.order_date)              AS first_order_date
#       MAX(o.order_date)              AS last_order_date
#   - ORDER BY: lifetime_value DESC
#
# A fájlt ide kell írni: f"{marts_dir}/mart_customer_ltv.sql"
# Fejléc komment: "-- mart_customer_ltv: ügyfélszintű lifetime value összesítés"
pass
      Megoldási útmutató – kattints a tippekért 
      mart_ltv_sql = textwrap.dedent("""\
    -- mart_customer_ltv: ügyfélszintű lifetime value összesítés
    -- Grain: 1 sor = 1 ügyfél összes megrendelésének összesítése
    SELECT
        c.customer_id,
        c.customer_name,
        c.city,
        c.country,
        COUNT(o.order_item_key)        AS total_orders,
        ROUND(SUM(o.total_amount), 2)  AS lifetime_value,    -- Teljes bevétel az ügyféltől
        ROUND(AVG(o.total_amount), 2)  AS avg_order_value,   -- Átlagos tételérték
        MIN(o.order_date)              AS first_order_date,   -- Első rendelés napja
        MAX(o.order_date)              AS last_order_date     -- Legutolsó rendelés napja
    FROM {{ ref('stg_orders') }}    o
    JOIN {{ ref('stg_customers') }} c ON o.customer_key = c.customer_id
    GROUP BY c.customer_id, c.customer_name, c.city, c.country
    ORDER BY lifetime_value DESC
    """)
with open(f"{marts_dir}/mart_customer_ltv.sql", "w") as f:
    f.write(mart_ltv_sql)
print("mart_customer_ltv.sql létrehozva.")

      Miért JOIN és nem LEFT JOIN? A stg_orders-ban csak olyan customer_key értékek vannak, amelyek szerepelnek a dim_customer-ben (az 1. feladatban FK kényszert alkalmaztunk). Így az INNER JOIN nem veszít el sorokat – és a szándékot is jobban kifejezi: csak azokat az ügyfeleket akarjuk megjeleníteni, akiknek van rendelésük.

    3. A hetedik cellában hozzuk létre az incremental mart modellt:

      # mart_daily_revenue_incremental.sql – inkrementális materializáció
# Nagy tábláknál csak az új sorokat dolgozza fel – nem futtatja újra az egész aggregációt
mart_incremental_sql = textwrap.dedent("""\
    {{ config(materialized='incremental', unique_key='full_date') }}

    -- mart_daily_revenue_incremental: napi bevétel – csak új napokat dolgoz fel újrafuttatáskor
    SELECT
        o.order_date                           AS full_date,
        COUNT(o.order_item_key)                AS item_count,
        ROUND(SUM(o.total_amount), 2)          AS total_revenue
    FROM {{ ref('stg_orders') }} o

    {% if is_incremental() %}
        -- is_incremental() = True, ha a tábla már létezik (nem full-refresh futás)
        -- Csak azokat a napokat dolgozzuk fel, amelyek újabbak az eddigi maximumnál
        WHERE o.order_date > (SELECT MAX(full_date) FROM {{ this }})
        -- {{ this }} = az aktuális tábla neve (main.mart_daily_revenue_incremental)
    {% endif %}

    GROUP BY o.order_date
    ORDER BY o.order_date
    """)
with open(f"{marts_dir}/mart_daily_revenue_incremental.sql", "w") as f:
    f.write(mart_incremental_sql)
print("mart_daily_revenue_incremental.sql létrehozva.")

      Hogyan működik az is_incremental()?

      Az első dbt run után a mart_daily_revenue_incremental tábla létezik. A következő futásnál a dbt felismeri, hogy a tábla már megvan, és is_incremental() értéke True lesz – a WHERE feltétel csak az utolsó betöltés óta érkezett sorokat dolgozza fel. A --full-refresh flag-gel kényszeríthetjük a teljes újraszámítást.
