# 4. feladat – Consumer groups

A Kafka consumer group-ok lehetővé teszik, hogy egy topicból több consumer párhuzamosan olvasson: minden partíciót egyszerre csak egy consumer kap a csoportból. Ez az alapja a horizontális skálázásnak. Ha egy consumer kiesik (crash), a Kafka automatikusan újraosztja a partíciókat a maradék tagok között (**rebalancing**). Ebben a feladatban két különböző consumer group-ot indítunk el párhuzamosan, és megfigyeljük, hogyan osztják el egymás között a partíciókat.

!!! note "Consumer group mechanizmusa"
    * Ha egy group-ban **kevesebb** consumer van, mint partíció, egyes consumerek több partíciót is kapnak.
    * Ha **több** consumer van, mint partíció, néhány consumer inaktív marad – feleslegesen foglal erőforrást.
    * Az **ideális arány**: `consumer szám == partíció szám` – minden consumer pontosan 1 partíciót kap.
    * Különböző `group_id`-val rendelkező csoportok **teljesen függetlenek**: mindkettő megkapja az összes üzenetet, saját offset-kezeléssel. Így pl. egy analytics és egy reporting rendszer egymástól függetlenül olvashatja ugyanazt a topicot.

    !!! note "Miért kell Lock a results listához?"
        Python szálak (threading.Thread) ugyanazt a memóriaterületet osztják meg. Ha két szál egyszerre próbál egy listához hozzáírni, adatsérülés (race condition) léphet fel. A `threading.Lock()` biztosítja, hogy egyszerre csak egy szál módosíthatja a listát. A `with results_lock:` blokk automatikusan feloldja a zárat, még kivétel esetén is.

    ??? tip "Megoldási útmutató – kattints a tippekért"
        ```python
        def run_consumer(group_id: str, consumer_id: str) -> None:
            consumer = KafkaConsumer(
                "orders",
                bootstrap_servers=BOOTSTRAP,
                group_id=group_id,                 # Ez azonosítja a csoportot
                auto_offset_reset="earliest",      # Az elejéről olvassa, ha nincs korábbi pozíció
                enable_auto_commit=True,           # Offset mentése automatikusan
                value_deserializer=lambda v: json.loads(v.decode("utf-8")),
                consumer_timeout_ms=4000,          # 4 mp után leáll, ha nincs új üzenet
            )
            n = 0
            for msg in consumer:
                if isinstance(msg.value, dict):
                    with results_lock:             # Zárolás a megosztott lista módosítása előtt
                        results.append({
                            "group_id":    group_id,
                            "consumer_id": consumer_id,
                            "partition":   msg.partition,
                            "offset":      msg.offset,
                        })
                    n += 1
            consumer.close()
            print(f"[{consumer_id}] Kész – {n} üzenet feldolgozva.")
        ```

28. Indítsuk el a két consumer group-ot párhuzamosan, és várjuk meg, amíg végeznek:

    ```python
    # threading.Thread: új szálat hoz létre, amely a target= függvényt futtatja az args= paraméterekkel
    threads = [
        threading.Thread(target=run_consumer, args=("group-analytics", "analytics-1")),
        threading.Thread(target=run_consumer, args=("group-reporting", "reporting-1")),
    ]
    for t in threads:
        t.start()   # Szál elindítása – azonnal visszatér, a szál a háttérben fut

    for t in threads:
        t.join()    # Megvárja, amíg az adott szál befejeződik – csak ezután lép tovább
    print(f"Összesen {len(results)} sor rögzítve.")
    ```

### Eredmény összesítése

29. Elemezzük, melyik consumer group melyik partíciókat olvasta:

    ```python
    # TODO: Hozzunk létre egy DataFrame-et a results listából.
    # Csoportosítsuk group_id és partition szerint, és számoljuk meg az üzenetek számát.
    # Írjuk ki a táblázatot.
    # Elvárt eredmény: mindkét group mind a 3 partícióból olvas (külön offsetekből).
    ```

    ??? tip "Megoldási útmutató – kattints a tippekért"
        ```python
        df = pd.DataFrame(results)

        # groupby több oszlop szerint is lehet – lista szintaxissal
        # size(): megszámolja az elemeket csoportonként
        # reset_index(): a csoportosítás oszlopait visszahozza normál oszlopokká
        summary = (
            df.groupby(["group_id", "partition"])
            .size()
            .reset_index(name="üzenet_szám")
        )
        print("Consumer group – partíció összesítő:")
        print(summary.to_string(index=False))
        ```

        **Mit várunk?** Mindkét `group_id` (`group-analytics` és `group-reporting`) mind a 3 partícióból olvas, de **különböző offset-értékektől** – mert mindkét group saját olvasási pozíciót tart nyilván.

    !!! note "Miért olvas mindkét group ugyanonnan?"
        Minden consumer group saját offset-pozíciót tart fenn a Kafka brokeren belül. Így ha két különböző group olvassa ugyanazt a topicot, mindkettő megkapja az összes üzenetet – függetlenül egymástól. Ez teszi lehetővé, hogy pl. egy analytics és egy billing rendszer egymástól teljesen függetlenül dolgozza fel ugyanazokat a rendelési eseményeket.
