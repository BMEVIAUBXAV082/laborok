# 4. feladat – dbt ELT pipeline

A dbt (data build tool) az ELT (Extract → Load → **Transform**) paradigma transzformációs motorja. Szemben az ETL-lel, ahol a transzformáció még betöltés előtt történik, az ELT esetén az adatokat először nyers formában töltjük be, majd az adatbázison belül transzformáljuk SQL segítségével. A dbt SQL modelleket Git-verziókövethető fájlokként kezeli, a `ref()` macro automatikusan feloldja a függőségi sorrendet, a `schema.yml` pedig beépített adatminőség-tesztelést biztosít.

!!! note "dbt alapfogalmak"
    * **`ref('model_name')`** – modell hivatkozás és függőség deklaráció egyben. A dbt ebből építi fel a végrehajtási DAG-ot: automatikusan tudja, melyik modellt kell először lefuttatni.
    * **Materializáció** – hogyan materializálódjon a modell az adatbázisban:
      - `view`: egy nézet, amely mindig friss adatot ad, de nem tárol adatot
      - `table`: előre kiszámolt tábla – gyorsabb lekérdezés, de tárhelyet igényel
      - `incremental`: csak az új/megváltozott sorokat dolgozza fel – nagy tábláknál hatékony
    * **Seed** – statikus CSV fájl, amelyet a `dbt seed` tölt be az adatbázisba (pl. referencia listák).
    * **Generic test** – YAML-ban deklarált adatminőség-ellenőrzés: `not_null`, `unique`, `accepted_values`, `relationships`.

    ```python
    # mart_daily_revenue.sql – napi bevétel összesítés

        GROUP BY o.order_date
        ORDER BY o.order_date
    ```

    ??? tip "Megoldási útmutató – kattints a tippekért"
        ```python
        mart_ltv_sql = textwrap.dedent("""\
            -- mart_customer_ltv: ügyfélszintű lifetime value összesítés
            -- Grain: 1 sor = 1 ügyfél összes megrendelésének összesítése
            SELECT
                c.customer_id,
                c.customer_name,
                c.city,
                c.country,
                COUNT(o.order_item_key)        AS total_orders,
                ROUND(SUM(o.total_amount), 2)  AS lifetime_value,    -- Teljes bevétel az ügyféltől
                ROUND(AVG(o.total_amount), 2)  AS avg_order_value,   -- Átlagos tételérték
                MIN(o.order_date)              AS first_order_date,   -- Első rendelés napja
                MAX(o.order_date)              AS last_order_date     -- Legutolsó rendelés napja
            FROM {{ ref('stg_orders') }}    o
            JOIN {{ ref('stg_customers') }} c ON o.customer_key = c.customer_id
            GROUP BY c.customer_id, c.customer_name, c.city, c.country
            ORDER BY lifetime_value DESC
            """)
        with open(f"{marts_dir}/mart_customer_ltv.sql", "w") as f:
            f.write(mart_ltv_sql)
        print("mart_customer_ltv.sql létrehozva.")
        ```

        **Miért JOIN és nem LEFT JOIN?** A `stg_orders`-ban csak olyan `customer_key` értékek vannak, amelyek szerepelnek a `dim_customer`-ben (az 1. feladatban FK kényszert alkalmaztunk). Így az INNER JOIN nem veszít el sorokat – és a szándékot is jobban kifejezi: csak azokat az ügyfeleket akarjuk megjeleníteni, akiknek van rendelésük.

37. A hetedik cellában hozzuk létre az incremental mart modellt:

    ```python
    # mart_daily_revenue_incremental.sql – inkrementális materializáció
    # Nagy tábláknál csak az új sorokat dolgozza fel – nem futtatja újra az egész aggregációt
    mart_incremental_sql = textwrap.dedent("""\
        {{ config(materialized='incremental', unique_key='full_date') }}

        -- mart_daily_revenue_incremental: napi bevétel – csak új napokat dolgoz fel újrafuttatáskor
        SELECT
            o.order_date                           AS full_date,
            COUNT(o.order_item_key)                AS item_count,
            ROUND(SUM(o.total_amount), 2)          AS total_revenue
        FROM {{ ref('stg_orders') }} o

        {% if is_incremental() %}
            -- is_incremental() = True, ha a tábla már létezik (nem full-refresh futás)
            -- Csak azokat a napokat dolgozzuk fel, amelyek újabbak az eddigi maximumnál
            WHERE o.order_date > (SELECT MAX(full_date) FROM {{ this }})
            -- {{ this }} = az aktuális tábla neve (main.mart_daily_revenue_incremental)
        {% endif %}

        GROUP BY o.order_date
        ORDER BY o.order_date
        """)
    with open(f"{marts_dir}/mart_daily_revenue_incremental.sql", "w") as f:
        f.write(mart_incremental_sql)
    print("mart_daily_revenue_incremental.sql létrehozva.")
    ```

    !!! note "Hogyan működik az is_incremental()?"
        Az első `dbt run` után a `mart_daily_revenue_incremental` tábla létezik. A következő futásnál a dbt felismeri, hogy a tábla már megvan, és `is_incremental()` értéke `True` lesz – a `WHERE` feltétel csak az utolsó betöltés óta érkezett sorokat dolgozza fel. A `--full-refresh` flag-gel kényszeríthetjük a teljes újraszámítást.

    ```python
        print("HIBA a docs generate során:", result.stderr[-2000:])
    else:
        print(result.stdout[-400:])

    # A catalog.json minden modell minden oszlopát tartalmazza
    with open(f"{PROJECT_DIR}/target/catalog.json") as f:
        catalog = json.load(f)

    print("dbt modellek és oszlopaik:")
    for node_name, node in catalog.get("nodes", {}).items():
        short_name = node_name.split(".")[-1]           # Pl. "mart_daily_revenue"
        cols       = list(node["columns"].keys())
        print(f"  {short_name}: {cols}")

    print("\nA lineage gráf megtekintéséhez futtasd egy új cellában:")
    print("  !dbt docs serve --project-dir /home/jovyan/work/labor04_dbt --host 0.0.0.0 --port 8081")
    print("Majd nyisd meg: http://localhost:8081")
    ```

    !!! tip "Lineage gráf böngészőben"
        A `docker-compose.yaml`-ban a `8081` port már fel van véve – ha a konténerek a régi compose fájllal indultak, előbb indítsd újra őket:
        ```powershell
        PS> docker compose down
        PS> docker compose up -d
        ```
        Ezután futtasd az alábbi parancsot egy **új** notebook cellában (ez blokkoló folyamat – `Interrupt Kernel`-lel állítható meg):
        ```python
        !dbt docs serve --project-dir /home/jovyan/work/labor04_dbt --host 0.0.0.0 --port 8081
        ```
        Nyisd meg: **[http://localhost:8081](http://localhost:8081)**

        A `--host 0.0.0.0` szükséges, mert alapból a szerver csak a konténeren belüli `127.0.0.1`-re köt – így a host gépről nem érhető el. A `0.0.0.0` minden hálózati interfészre figyel, és a Docker port forwarding ezen keresztül tud működni.
